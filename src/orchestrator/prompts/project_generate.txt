Generate ALL of the following files for a proper Python project:

   - `src/{project_name}/main.py` — main entry point
   - `src/{project_name}/models.py` — data models (if applicable)

**QUALITY STANDARDS:**
1. PEP 8 compliance with Google-style docstrings on ALL functions and classes
2. Type hints on every function signature
3. Proper `__init__.py` with `__all__` exports for public API
4. Error handling with custom exceptions — no bare `raise` or generic `Exception`
5. Logging via `logging` module — no `print()` statements
6. Configuration via environment variables or pydantic Settings
7. All imports at the top of files, organized stdlib → third-party → local
8. No hardcoded secrets, paths, or magic numbers
9. Functions under 30 lines, classes under 150 lines
10. Proper separation of concerns (models, services, routes for web projects)

**INTER-MODULE IMPORTS:**
All modules must use correct relative or absolute imports. For example:
- `from {project_name}.models import UserModel`
- `from {project_name}.config import settings`
- `from {project_name}.exceptions import NotFoundError`

IMPORTANT:
- Every Python file must be syntactically valid
- **Every name you import must be a file you also generate.** If you
  write ``from calculator_app.exceptions import NotFoundError`` you
  MUST emit a corresponding ``calculator_app/exceptions.py`` in the
  ``files`` array. A common failure pattern is to write a single
  ``calculator.py`` that imports from siblings you forgot to generate —
  every such broken import will halt test collection. Make sure the
  set of files you emit is **closed under imports**: for every
  ``from x.y.z import ...`` mention, ``x/y/z.py`` (or ``x/y/z/__init__.py``)
  is in your output.
- All imports must resolve correctly within the generated project
- The project must be installable via `pip install -e .`
